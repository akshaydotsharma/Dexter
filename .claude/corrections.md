# Project Corrections

Project-specific corrections for Dexter. See `~/.claude/rules/self-corrections.md` for the protocol.

- **2026-05-09 (SIM_QA_BEFORE_DEVICE_HANDOFF)**: Before asking the user to install or verify a build on their phone, run the change in the iPhone simulator (`xcrun simctl boot` + `xcodebuild ... -destination 'platform=iOS Simulator'` + `simctl install` + `simctl launch` with `SIMCTL_CHILD_LAUNCH_SECTION=<section>` if a deep-link env var exists) and capture screenshots of the affected surfaces. Read the screenshots back to confirm the change is visible. Only then surface to the user. "BUILD SUCCEEDED" is not QA — it never was — and shipping over OTA without UI proof wastes the user's install cycle and trust. The simulator path takes ~30 s and is the cheapest insurance available.
- **2026-05-09 (RENAME_FULL_GREP_SWEEP)**: Any brand or string rename (e.g. "Deks" -> "Dexter") MUST start with a repo-wide `grep -rn '<old>'` across `--include="*.swift" --include="*.plist" --include="*.yml"` BEFORE the first build. Hardcoded user-facing strings live well outside the obvious view file: in `Intents/*.swift` (App Intent titles, descriptions, parameter summaries), `Services/*.swift` (permission error messages), Shortcut provider docs, and `project.yml` (xcodegen regenerates `Info.plist` from this — editing the plist directly is reverted on next `xcodegen generate`). Skipping the sweep means a half-renamed app reaches the user.
- **2026-05-09 (XCODEGEN_PLIST_AUTHORITY)**: `mobile/PersonalDashboard/Info.plist` is regenerated from `mobile/project.yml` on every `xcodegen generate`. Edit `project.yml` (under `targets.PersonalDashboard.info.properties`) for any plist change — `CFBundleDisplayName`, usage description strings, Bonjour services, etc. Edits to `Info.plist` directly will be silently reverted on the next regen.
- **2026-05-09 (DEVICECTL_VERBOSE_RETRY)**: When `xcrun devicectl device install app` fails with `CoreDeviceError 1011 "unable to locate a device matching the requested device identifier"` despite the device showing as `available (paired)` in `devicectl list devices`, retry with `--verbose`. The verbose flag triggers a fresh tunnel acquisition + DDI mount path that frequently succeeds where the bare command failed. Observed working on iPhone 16 Pro Max + iOS 26 + free personal-team signing. Try this BEFORE falling back to the OTA URL.
- **2026-05-10 (AUTOSHIP_TO_PHONE)**: Once an iOS feature compiles cleanly, ship to the phone autonomously via the `build-to-phone` skill — do NOT pause to ask permission first. Order is: implement → static build passes → ship to phone → user does device QA → commit → push → PR. The user has already authorised this loop globally for this repo; asking each time wastes a turn. Reason: user direction on 2026-05-10. How to apply: invoke `build-to-phone` skill immediately after a successful `xcodebuild ... build`, before reporting the feature as ready. The OTA + devicectl install IS the next step; treat it as part of the task, not a follow-up question.
- **2026-05-22 (NO_QA_EVIDENCE_ON_TICKETS)**: Dexter is a personal solo-built app — skip the "comment screenshots + ticked AC onto the GitHub issue" step from `~/.claude/rules/issue-resolution-workflow.md`. The user explicitly opted out on 2026-05-22 ("we don't need [QA evidence] over here"). Issue close-out via `Closes #N` in the PR is sufficient; do NOT volunteer to add retroactive QA comments and do NOT ask whether to add them. How to apply: when shipping a [Feature]/[Bug]/[Enhancement] in this repo, raise the ticket → implement → ship to phone → user does device QA verbally → commit → PR with `Closes #N` → merge. Stop there.
